## 1. 地图、网格与舰船的尺度关系

### 三套尺寸

Jutland 地图关卡同时存在三套尺寸，必须分别计算：

- **源图像素尺寸**：`resources/images/map/abbrs/<source>.png` 的宽高。
- **地图网格尺寸**：`resources/maps/<map>.map` 的行列数。
- **游戏逻辑尺寸**：地图格数乘 `constants.MapBlockSize`。当前 `MapBlockSize = 128`。

地图加载器按以下方式处理源图：

```text
sourceBlockPixels = sourceImageWidth / mapGridWidth
renderScale = MapBlockSize / sourceBlockPixels
logicalMapWidth = mapGridWidth * MapBlockSize
```

因此源图为 8192 像素并不自动表示游戏宽度为 8192 米。128 格地图始终得到 `128 × 128 = 16384` 个逻辑长度单位。

舰船的 `length`、`width` 以米配置，普通舰船顶视图最终按对应像素长度绘制。地图逻辑尺寸与舰长可以直接比较。

### 地标比值法

选择真实尺寸可靠的地标，计算：

```text
historicalRatio = realLandmarkLength / realShipLength
gameRatio = landmarkCells * MapBlockSize / configuredShipLength
scaleError = gameRatio / historicalRatio
```

解释：

- `scaleError ≈ 1`：相对尺度基本一致。
- 长、宽误差都接近同一系数：整体缩放错误。
- 长、宽误差差异明显：源图可能被非等比拉伸。
- 误差小于约 10%：通常可由岸线简化、资料口径或网格量化解释。
- 误差超过约 15%：部署前必须报告；不得继续宣称精确历史尺度。

优先选择岛屿、跑道、干船坞、防波堤等边界明确的地标。不要用树木、建筑群或透视航拍中的模糊距离。

### 修复策略

按风险从低到高选择：

1. 整体缩放 PNG 中的地貌并同步重建 `.map`，再重算任务坐标。
2. 重新制作覆盖正确真实范围的底图和网格。
3. 引入每地图独立的米/格尺度，并完整修改绘制、碰撞、移动、射程和路径逻辑。

不要只缩放舰船贴图。视觉放大而几何尺寸不变会导致炮弹命中、碰撞、搁浅判断和甲板动画错位。

## 2. 历史资料与快照时间

### 证据优先级

按以下顺序取证：

1. 同时刻战斗报告、航海日志、港务泊位记录。
2. 官方海军舰位图、损害报告附图和同时期垂直航拍。
3. 舰史、官方博物馆或政府历史页面。
4. 有明确来源注释的二手研究。
5. 普通百科、论坛或无来源示意图，只用于发现线索。

对每个关键结论记录：来源、时间、舰位、舰首方向、是否估算。不同资料冲突时优先采用更接近目标时刻的第一手记录，并在注释或交付说明中写明取舍。

### 固定一个快照

关卡必须固定一个时间点，而不是笼统使用某一天：

- 已离开泊位的舰船按该时刻实际航道位置部署。
- 已受损、倾覆或移动的舰船不能与袭击开始前阵位混用。
- 进攻编队若按玩法要求移入地图，区分“历史编成”与“游戏入口位置”。

## 3. 坐标和朝向换算

### 坐标映射

简单无旋转参考图可以按比例换算：

```text
mapX = imageX / imageWidth * gridWidth
mapY = imageY / imageHeight * gridHeight
```

但只有在参考图和游戏地图岸线完全一致时才可使用。遇到裁切、旋转、透视或艺术化岸线时：

1. 选择至少三个不共线控制点。
2. 求仿射映射或分区域映射。
3. 用未参与求解的第四个地标检查误差。
4. 再把泊位和舰队中心投影到网格。

先确定编队中心和泊位，再分配单舰位置。整数格不足以表达真实并靠宽度时，优先保持相对关系和可读性。

### 舰首方向

游戏约定：

| rotation | 舰首方向 |
|---:|---|
| 0 | 北/屏幕上 |
| 90 | 东/屏幕右 |
| 180 | 南/屏幕下 |
| 270 | 西/屏幕左 |

东北约 `45°`，东南约 `135°`，西南约 `225°`，西北约 `315°`。

照片或图中的黄色箭头可能表示攻击方向、航迹或视线，不一定表示舰首。优先使用战斗报告中的 true heading；只有缺少记录时才从泊位轴线推断，并明确写“约”。

## 4. 舰船替代决策

按以下顺序选择：

1. 同名舰。
2. 同级姊妹舰。
3. 近缘级或同系列改型。
4. 同国家、同类别、排水量和长度接近的舰。
5. 只有用户明确接受时，使用其他国家或跨年代近似。

同类别近似时比较：标准/满载排水量口径、舰长、服役年代、航速、武器角色和防护。不要只按名称或视觉轮廓选择。

每艘替代舰使用以下注释格式：

```json5
// Arizona BB-39 -> pennsylvania：同属 Pennsylvania 级；F-7 内侧，舰首约西南。
{
  name: "pennsylvania",
  pos: [68, 61],
  rotation: 225,
  belongPlayer: "HA"
},
```

即使多个历史舰共享同一个资源，也要分别保留历史身份注释。

## 5. 港区粗网格处理

`.map` 中 `.`、`O`、`S` 可航行，`C`、`L` 不可航行。128 米级网格无法表达真实码头间隙和并靠舰宽，常见问题包括：

- 舰体中心位于水格，但舰首或舰尾进入岸格。
- 干船坞、窄港池和栈桥之间的水面被合并为岸格。
- 多舰并靠需要相邻整数格，视觉间距大于历史间距。

处理原则：

1. 中心点必须在可航行格。
2. 检查舰长矩形的首尾和四角；压岸作为警告逐项审阅。
3. 移到最近水格时保持泊位的纵向顺序。
4. 并靠舰保持内外舷顺序，不因水格不足前后颠倒。
5. 无法兼顾时优先可玩性与关系正确，并在关卡注释和交付说明中披露偏移。

## 6. 任务配置与验证清单

### 配置

- 任务名唯一，`mapName` 存在。
- 四种显示名和描述完整。
- `initCameraPos` 在地图范围内。
- `initShips` 中每个资源名存在于 `configs/ships.json5`。
- 阵营只使用 `HA`、`HB`、`CA`、`CB`。
- `rotation` 在 `[0, 360)`。
- 无需求时 `initReinforcePoints`、`initOilPlatforms` 使用空数组。
- `maxShipCount` 按现有玩法语义设置，不把它未经确认地当作双方总舰数。

### 验证

1. 运行 skill 自带验证脚本，修复引用、越界和中心压岸错误。
2. 检查舰体边缘警告和重复坐标警告。
3. 统计双方舰数及实际资源类型，与用户要求核对。
4. `git diff --check -- configs/missions.json5`。
5. 运行相关测试、`go test ./...`、`go build ./...`。
6. 若环境可启动窗口，手动检查地图比例、舰体重叠、首向、初始镜头和 AI 可达路径。

## 7. 珍珠港案例经验

珍珠港素材说明了为什么比例审计必须先于布阵：

- 当前 PNG 为 `8192×10240`，地图为 `128×160`，横纵两个方向的源图每格均为 `64` 像素。
- 运行时每格统一绘制为 `128` 像素，源地貌被放大约 2 倍。
- 福特岛在网格中的长轴约 `35.9` 格，逻辑长度约 `35.9×128=4.59 km`。
- 福特岛真实长约 `2.41 km`；Nevada 舰长 `178 m`。
- 历史比例约 `13.6` 个 Nevada 舰长，游戏比例约 `25.8` 个，误差约 `1.9×`。

源图中的福特岛像素长度本身接近真实米制，错误来自源图格尺寸与固定 `MapBlockSize` 的组合。正确结论是先修地图及网格并重算舰位，而不是静默把舰船贴图放大。
