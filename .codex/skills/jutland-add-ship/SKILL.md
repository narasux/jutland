# Jutland 添加战舰

## 核心规则

- 在 Jutland 仓库根目录工作。先阅读 `AGENTS.md`，执行 `git status --short`，保留所有无关的用户改动。
- 不重新生成、不重绘、不风格化、不替换用户提供的蓝图素材。只允许裁剪、拆分、清理透明背景和等比例缩放。
- 所有图片缩放都必须锁定裁剪后素材的长宽比。禁止为匹配 `ships.json5` 的 `length`、`width` 或目标像素尺寸而分别缩放宽度和高度，也禁止横向压缩、横向拉伸、纵向压缩或纵向拉伸。
- 如果视图边界、舰体轮廓、栏杆空隙、桅杆/索具、迷彩边缘、背景与舰体分离关系不确定，先停下来询问用户。
- 每次尝试更激进的透明化或边缘清理前，先在同目录备份当前 PNG，例如 `<name>.before-rail-cleanup.png` 或 `<name>.before-aggressive-cleanup.png`。
- 保持配置和资源引用链一致：同一个战舰 `name` 必须同时匹配 PNG 文件名和所有配置引用。
- 优先沿用现有战舰、武器、飞机、发射器配置模式。只有现有配置无法合理表达新战舰时，才新增武器族配置。
- 当战舰使用特定 Mark 型号武器（如 Mark 16 速射主炮、Mark 22 防空炮）而现有 `guns.json5` 只有通用型号或不存在对应 Mark 条目时，必须新增独立武器条目并在舰船配置中引用；详见下方"新增 Mark 专用武器"章节。
- 本仓库默认只做编译/构建验证；除非用户明确要求，不主动跑测试。

## 仓库目标文件

- 战舰配置：`configs/ships.json5`
- 图鉴引用：`configs/references.json5`
- 测试任务：`configs/missions.json5` 中 `name: "TestAll"` 的任务
- 战舰图片：
  - `resources/images/ships/top/<type>/<name>.png`
  - `resources/images/ships/side/<type>/<name>.png`
- 现有战舰图片加载器在 `pkg/resources/images/ship/ship.go` 中扫描固定舰种目录。战舰 `type` 必须匹配已存在的资源目录，例如 `aircraft_carrier`、`battleship`、`cruiser`、`destroyer`、`frigate`、`torpedo_boat`、`cargo`、`hospital`。

## 新增 Mark 专用武器

当战舰历史武装包含特定 Mark 型号，而 `configs/guns.json5` 中仅有通用型号或无对应条目时，必须新增武器。命名规则：`country/caliber/length/barrel_count/MKtype`（例如 `US/203/55/3/MK16`）。

### 流程

1. 在 `configs/guns.json5` 中找到最接近的通用型号，在其条目闭合 `},` 后插入新条目。
2. 新条目的 `bulletName` 复用现有弹药即可，除非炮弹性能显著不同；Mark 差异主要体现在装填时间、散布和射程。
3. 派生数值参照历史性能：
   - 装填时间：Mark 16（10 发/分）→ 5.5s，对比通用型（8.5 发/分）→ 7s
   - 散布：同口径速射型通常散布收窄 10%~15%
   - 双联装 vs 单装：`bulletCount` 设为 2，装填时间约为单装的一半（双管交替射击）
4. 双联装 AA 炮（如 `US/76/50/2/MK22`）：每一条目代表一座双联装位，12 个条目 = 12 座 × 2 管 = 24 管。船体总炮管数 = 条目数 × `bulletCount`。
5. 更新 `configs/ships.json5` 中新舰的武器引用名称。
6. 更新 `configs/references.json5` 中 `armaments` 描述，标注 Mark 型号。

### 命名对照

- 单装无 Mark：`US/203/55/3`（通用型）
- 单装带 Mark：`US/127/38/MK12`
- 双联带 Mark：`US/127/38/2/MK12`
- 三联带 Mark：`US/203/55/3/MK16`
- 双联 AA 炮 Mark：`US/76/50/2/MK22`（仅 ≥ 40mm）

### 注意事项

- 不主动修改已有战舰的武器引用；新增的 Mark 条目仅供新舰使用。
- 口径 ≥ 40mm 的防空炮需要 Mark 标识（如 `US/76/50/2/MK22`），为后续性能差异化预留空间。口径 < 40mm 的小口径火炮（20mm、25mm 等）不区分 Mark 型号，统一使用无 Mark 后缀的通用条目。
- 如果当前蓝图未显示某些武器（如 20mm），但历史资料明确列出，仍需配置并标注"图片可见性待核实"。

## 混合舰种与航空战舰

- 航空战列舰、航空巡洋舰等混合舰优先按舰体来源和主要水面作战身份选择现有 `type`，航空能力通过 `aircraft` 独立配置。现有数据模型能够表达"`battleship` + 舰载机"时，不要仅为展示名称新增舰种目录、`ShipType` 或资源加载分支。
- 只有当新舰种需要独立的 UI 分类、增援路由、资源目录或玩法规则，且现有 `type` 无法表达时，才扩展 Go 枚举和加载器；同时检查类型展示、任务分组和所有按舰种判断的逻辑。
- 混合舰俯视图存在航空甲板、弹射器或舷外平台时，配置 `width` 仍使用实际舰体宽度，图片则始终保留原素材比例和外伸结构，不按 `width*4` 强制压窄。
- 历史机型在 `configs/planes.json5` 中不存在时，优先复用同阵营、同任务角色的现有飞机，并在图鉴中标注"游戏近似"；除非用户明确要求，不因添加一艘船同时新增飞机、机载武器和素材链。
- 弹射机、一次性起飞或历史上无法直接着舰回收的航空运用，与当前自动返航回收模型存在差异。默认保留现有全局机制并在最终说明中记录近似，不为单舰修改通用回收逻辑。
- `TestAll` 的增援列表和初始舰布局按配置中的主 `type` 分组；航空战列舰仍加入战列舰列表，而不是同时重复加入航母列表。

## 工作流程

1. 检查现有模式。
   - 阅读 `configs/ships.json5`、`configs/references.json5` 和 `configs/missions.json5` 的 `TestAll` 区段。
   - 按 `type`、海军阵营、时代、吨位和定位选择 1-3 艘相似舰作为平衡基准，用于推导费用、减伤、加速度和转向速度。

2. 确认用户输入。
   - 询问或推断内部名 `name`、展示名 `displayName`、战舰 `type`、`typeAbbr`、舰船资料链接、蓝图/美术来源链接和作者。
   - 如果用户只提供常用舰名，先提出 snake_case 格式的 `name` 建议，并在写文件前让用户确认。

3. 拆分并清理图片。
   - 从用户提供的蓝图中提取侧视图和俯视图。除非用户明确要求保留正视图，否则不处理正视图。
   - 先裁剪，再透明化，再缩放；缩放后还要再次检查并清理边缘白点，因为白底图缩放会产生新的近白色抗锯齿残留。
   - 缩放时只能指定一个目标轴或统一缩放系数，让另一个轴按素材比例自动计算。不得同时指定不符合原比例的目标宽高。
   - 将非战舰背景透明化。明显的栏杆、桅杆、吊臂、上层建筑间隙需要处理；边界不确定时交给用户确认或手工处理。
   - 蓝图分析：当模型无法识别图片内容时，使用 Python PIL 逐像素分析定位视图边界：
     - 先用 `sips -g pixelWidth -g pixelHeight` 获取尺寸。
     - 对大尺寸蓝图（>2000px），使用步进采样（step=10~20）避免超时。
     - 行密度扫描（统计每行非白像素数）找到水平视图分界；列密度扫描找到垂直分割及文本标签区域。
     - 内容边界判定阈值 `r/g/b < 245`，密度阈值 >5（步进采样）或 >20（逐行采样）。
     - 俯视图左侧边缘可能有文本标注，需检查暗像素密度（>20% 暗像素区域）并排除。
   - **剪裁禁止过度紧贴**：侧视图的桅杆、雷达天线等细长结构在密度采样时极易被漏检（单像素宽的线在步进采样中可能完全跳过）。裁剪纵向范围必须在检测到的最靠上内容边界上方留出至少 50px（原始分辨率）的余量；宁可多留透明边距，也不截断桅杆或天线。同理，俯视图的左右舷和舰首舰尾也要适当放宽。
   - 将透明 PNG 保存到对应 `top` 和 `side` 舰种目录。

4. 根据资料调整图片尺寸。
   - 优先读取用户提供的资料链接。如果无法联网或页面不可访问，向用户索要长度、宽度/型宽、排水量、速度和武装信息。
   - 真实尺寸四舍五入取整后写入配置：`length` 和 `width`。
   - `configs/ships.json5` 的 `width` 使用资料中的实际舰体宽度、型宽或船体宽度，不要用飞行甲板外廓宽度替代。百科中的"宽度/型宽"通常不是航母飞行甲板宽度。
   - 配置中的 `length` 和 `width` 只描述游戏内舰体几何尺寸，不是 PNG 两个像素轴的强制目标，也不能用于改变图片素材比例。
   - 俯视图可以把纵向长度缩放到约 `length*4`，但必须等比例计算图片宽度；即使图片视觉宽度与 `width*4` 不一致，也不得压缩或拉伸。
   - 侧视图可以把横向宽度缩放到约 `length*4`，但必须等比例计算图片高度；资料中的舰高或其他尺寸也不能作为非等比例缩放依据。
   - 若同类资源尺寸与等比例缩放结果冲突，优先尊重素材长宽比，只调整统一缩放倍数或保留透明边距，不修改舰体比例。
   - 定稿前用 `sips -g pixelWidth -g pixelHeight` 对比现有同类图片尺寸。

## 图片透明化和边缘清理

- 优先使用"从画布边缘泛洪"的方式删除背景：只让透明像素和近白色背景像素从图片外部向内连通扩散，再把这些连通像素设为透明。这样不会误删飞行甲板中间的白色跑道线、识别线和标识。
- 背景阈值从保守开始，例如 `r/g/b >= 245`；缩放后出现白边或白点时，再放宽到 `>= 210` 左右，并要求 RGB 差值较小（低饱和近白色），避免删除浅灰舰体。
- 对栏杆、索具、吊臂围起来的封闭白底，外部泛洪不会处理。需要单独做连通组件清理：
  - 先统计近白色组件的位置、大小和包围盒，确认残留集中在哪些区域。
  - 俯视图优先只处理左右舷外缘带，保留中央飞行甲板线。
  - 侧视图优先处理舰体上方的索具、栏杆和外轮廓区域，避免清理红色船底、船体高光和内部浅灰结构。
  - 从小组件、外缘组件开始；如果仍不干净，再备份后扩大区域或放宽阈值。
- 俯视图的栏杆、安全网、舷外网格等细线区域要优先在缩放前的高分辨率图上清理底色，然后再等比例缩放：
  - 不要在缩放后的低分辨率 PNG 上用低阈值（例如 `avg >= 135` 或 `>= 150`）清理栅栏底色；缩放后的黑色栅栏抗锯齿会变成中灰，低阈值会误删栅栏线，形成成片空白。
  - 对安全网格底色只删除高分辨率图中低饱和的亮色填充，例如 `avg >= 225~235` 且 RGB 差值较小；保留深灰、黑色和半透明抗锯齿像素。
  - 栅栏清理区域必须严格限制在舰体外侧安全网带，避免覆盖飞行甲板边线、跑道虚线、识别线和舰体外轮廓。
  - 如果用户明确表示可以自行回填甲板标线，可以采用更宽的高分辨率清理区来换取更干净的安全网底色；这种方案必须在最终说明中标明"可能影响/已影响哪些甲板标线，需要手工回填"，不能当作默认方案。
  - 如果低分辨率清理已经误删栅栏或切到甲板，优先从原始蓝图重新裁切生成该视图，不要在损坏 PNG 上继续修补。
  - 缩放带透明边缘的图像时，优先使用预乘 alpha 或等效流程，减少隐藏白色 RGB 在透明边缘产生白边。
- 每次处理后用 `view_image` 预览，并记录清理掉的组件数或像素数。若发现误删结构，恢复最近的备份后收紧区域或阈值。
- 对透明 PNG 的预览不要只看默认白底显示；必须额外合成到黑底或高对比色背景（例如洋红）检查真实透明效果，尤其是栅栏、安全网和索具区域。
- 如果用户指出局部区域仍有白点，优先对该视图重新分析近白色连通组件，不要手工逐像素猜。

5. 更新 `configs/ships.json5`。
   - 将新舰放到同舰种/同阵营的相邻分组附近。
   - `totalHP` 优先使用满载排水量；资料只有标准排水量时，使用最可靠的吨位并在回复中说明。
   - `maxSpeed` 使用资料中的最高航速，单位为节。
   - `fundsCost`、`timeCost`、`horizontalDamageReduction`、`verticalDamageReduction`、`acceleration`、`rotateSpeed` 从相似现有战舰推导。
   - 根据资料和图片可见武装配置主炮、副炮、防空炮、鱼雷、火箭和舰载机。尽量复用已有 `guns.json5`、`torpedo_launchers.json5`、`rocket_launchers.json5`、`planes.json5` 中的名称。每个武器挂点根据图片和同类配置设置 `posPercent` 与射界。
   - 航空母舰只在项目中已有对应飞机配置时添加 `aircraft` 编组。

6. 更新 `configs/references.json5`。
   - 添加同名 `name` 条目。
   - `specs` 包含类型、吨位、速度、费用和减伤。
   - `armaments` 简洁概括已配置的武器和舰载机。
   - `description` 根据资料写一段简短历史介绍。
   - `author` 使用蓝图/美术作者；未知则写 `未知`。
   - `links` 同时包含蓝图/美术来源和舰船资料来源。

7. 更新 `configs/missions.json5` 的 `TestAll`。
   - 按战舰 `type` 将新舰加入对应 `providedShipNames` 列表。
   - 在 `initShips` 中新增一艘己方 `HA` 初始舰，延续现有同舰种网格布局，避免与附近舰船重叠。
   - 除非用户明确要求，不修改其他任务。

8. 完成前验证。
   - 确认俯视图和侧视图 PNG 都存在，并确认每个视图相对裁剪后素材保持原始长宽比；允许因像素取整产生最多 1 像素误差。
   - 确认图片处理没有为了匹配配置 `width` 或同类图片尺寸而使用非等比例缩放。
   - 确认 `ships.json5`、`references.json5`、图片文件名和 `TestAll` 都使用同一个 `name`。
   - 确认引用到的武器、发射器、飞机和弹药配置都存在。
   - 运行 `GOCACHE=/tmp/go-build go build -o /dev/null ./pkg/...`。
   - 最终回复中明确说明未验证的图片清理区域或资料假设。
