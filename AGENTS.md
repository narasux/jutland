# Jutland Agent 指南

本文件用于指导编码 agent 在本仓库中工作。Jutland 是一个基于 Go 与 Ebiten 的 2D 海战即时策略游戏，代码、配置、资源三者强耦合；改动时必须同时考虑运行时资源加载、JSON5 配置兼容性和游戏循环性能。

## 工作原则

参考 Andrej Karpathy 风格编码指南，所有非琐碎任务遵循以下原则：

1. **编码前思考**
   - 不确定需求、行为或配置含义时，先说明假设；有高风险歧义时先询问。
   - 给出必要权衡，尤其是玩法平衡、性能、资源格式、兼容性相关改动。

2. **简洁优先**
   - 用最少代码完成目标，不为一次性逻辑新增抽象。
   - 不添加用户未要求的功能、配置项或“以后可能用到”的扩展点。
   - 如果现有包内已有相近模式，优先沿用。

3. **精准修改**
   - 只修改完成当前任务必须触碰的文件。
   - 不顺手重构相邻代码、格式化无关文件、清理既有死代码或移动资源。
   - 不删除、替换或压缩现有素材文件，除非任务明确要求。

4. **目标驱动执行**
   - 把任务转化为可验证的完成标准。
   - 修复 bug 时优先补充或更新能复现问题的测试；无法测试的游戏表现改动，需要说明手动验证方式。

## 项目结构

- `main.go`：程序入口，加载游戏设置、配置 Ebiten，并启动游戏。
- `pkg/game/`：游戏主状态机、菜单、输入、绘制入口。
- `pkg/mission/`：任务运行核心，包括状态、控制器、对象、指令、绘制、Hacker 终端。
- `pkg/mission/object/initialize/`：从 `configs/*.json5` 初始化全局对象映射，加载顺序敏感。
- `pkg/resources/`：图片、字体、音频等资源加载封装；通常依赖 `resources/` 下的实际文件。
- `pkg/utils/`：几何、布局、网格、颜色等通用工具，适合添加单元测试。
- `configs/`：游戏配置，使用 JSON5。修改配置字段时同时检查对应 Go struct、初始化逻辑和 `configs/Readme.md`。
- `resources/`：运行时素材。路径、文件名、目录结构会影响加载逻辑。

## 开发命令

- 整理依赖：`make tidy`
- 格式化：`make fmt`
- 单元测试：`make test`
- 构建：`make build`
- 运行：`make build && ./jutland`

注意：

- 以 `go.mod` 中的 Go 版本为准。
- Ebiten 通常需要 CGO 和本机图形/音频环境。CI 或无 GUI 环境中，优先运行 `go test ./...`；不要假设可以启动游戏窗口。
- `make fmt` 依赖 `golines` 和 `gofumpt`，如果本机缺失，不要改用大范围手动格式化；说明未能执行即可。

## Go 编码约定

- 遵循现有包结构和命名方式，不随意引入新顶层包。
- 保持中文注释风格；只在复杂游戏逻辑、坐标换算、物理/几何计算处添加必要注释。
- 错误处理沿用现有模式：启动期资源或配置缺失多使用 `log.Fatal`，运行中可恢复错误应返回或记录。
- 不在热路径中引入明显额外分配、反射或复杂遍历，尤其是 `Update`、绘制、碰撞、武器开火、AI 控制相关代码。
- 对浮点计算使用容差断言，参考 `pkg/utils/geometry/geometry_test.go`。

## 配置与资源规则

- `configs/*.json5` 允许注释，保持现有字段风格和命名。
- 新增武器、舰船、飞机、地图或任务时，检查引用链是否完整：
  - 弹药 -> 武器/发射器 -> 舰船/飞机 -> 任务/地图。
  - 资源文件名需要与配置中的对象名称或加载逻辑匹配。
- 修改配置单位时要追踪初始化阶段的缩放换算，例如射程、速度、装填时间等。
- 图片、音频、字体和地图素材属于项目运行资源；不要做无关压缩、重命名或批量迁移。
- 项目 README 声明素材仅供学习且不得商业使用；涉及发布、打包、素材来源时保留该约束。

## 测试与验证

优先选择能在当前环境稳定运行的验证：

1. 纯工具函数或状态计算：添加/更新单元测试，然后运行 `go test ./...` 或更小范围测试。
2. 配置解析或资源映射：运行相关测试；如无测试，至少运行 `make build` 验证编译和初始化引用。
3. 游戏交互、绘制、音频：运行 `make build`；如本机可开窗口，再手动验证关键路径。
4. 格式化：改 Go 代码后运行 `make fmt`，或在工具缺失时运行 `gofmt`/说明限制。

完成前不要声称“通过”某项验证，除非实际执行并确认结果。

## 任务处理建议

- 小改动：直接定位相关文件，最小修改，运行最窄验证。
- Bug 修复：先复现或写测试，再改实现，最后验证。
- 玩法/平衡改动：明确影响的配置、单位换算和预期体验，避免同时改代码与大量数值导致难以回溯。
- 重构：只有在用户明确要求或现有结构阻碍目标时进行；保持行为等价并用测试或构建验证。
- 资源相关任务：先确认路径和加载逻辑，再增删资源；不要自动清理 `.DS_Store` 或其他用户已有未跟踪文件，除非用户要求。

## Git 与工作区

- 当前工作区可能包含用户未提交改动。开始修改前查看 `git status --short`。
- 不回滚、不删除、不覆盖自己没有创建的改动。
- 遇到无关脏文件时忽略；遇到会影响当前任务的脏文件时，先读懂并兼容它们。
- 不使用 `git reset --hard`、`git checkout -- <file>` 等破坏性命令，除非用户明确要求。
